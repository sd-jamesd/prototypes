DAM Event Taxonomy

The capture contract for the DAM event store. Defines the envelope, the event type registry, and the payload per event. This is the write side only — the audit trail, activity views, and reporting surfaces are read paths built over this stream and are specified elsewhere.

v0.1 — draft for build Owner: Product Implements: DAM event capture (parent)

Principle

Capture and display are separate concerns. Record everything append-only, filter at read time. A display requirement never constrains the schema, and a display decision that has not been made yet never blocks capture.

One store, many reads. The asset audit trail is one filtered read of this stream. An activity dashboard view is a different read. Compliance export is a third. Building it the other way round — a log per feature — produces three overlapping stores that cannot answer a question spanning any two of them.

  • Events are immutable. Never updated, never deleted, corrected only by a subsequent compensating event.
  • The store is separate from the asset index. Confirmed in the metadata architecture: the assets index does not carry audit or event data.
  • Capture breadth is cheap now and impossible later. When uncertain whether an event is worth recording, record it — the retention policy is the place to manage volume, not the schema.

What cannot be retrofitted

Three items have no recovery path. Everything else in this document can start late and accept a gap in the data. These cannot.

1. Provenance at promotion or upload. Originating workflow, form, approval chain, and instigating actor exist only at the moment the asset enters the DAM. An asset that arrives before capture is live is permanently missing its lineage — which is the workflow-provenance claim the product is positioned on.

2. The correlation key. Events written without unity_context_id are permanently unjoinable to workflow events. Retrofitting means re-deriving relationships from data that was never recorded.

3. Mint-endpoint logging. CDN edge logs give bytes, not meaning — by design, since the CDN makes no rights decisions. If delivery ships without structured mint logging, download reporting has no source.

Present on every event regardless of category. Category-specific payload sits on top of this.

Naming convention

  • domain.action, lower case, past tense: rights.changed, asset.downloaded.
  • One event type per meaningful state transition, not per field written. A bulk edit touching six attributes emits one attributes.changed with all six in the payload, not six events.
  • Change events carry before and after in the payload. Absence of before means creation.
  • New types are additive. Existing type semantics never change — that would silently reinterpret history.

Category 1 — Provenance

One-shot, written once at the moment an asset enters the DAM. Highest priority in the whole specification: unrecoverable after the fact.

Event typePayloadTimingRetention
asset.promoted Originating workflow (ID, name, type) · originating form ID · triggering action (approval complete / manual push / other) · instigating user · promotion timestamp · approval chain (per reviewer: identity, decision, timestamp, review stage) · source rendition reference · rights state at ingest · auto-attributed metadata carried in. Review stage stored as data, never a fixed label. One-shot · unrecoverable audit
asset.uploaded Upload origin (direct / draft area / bulk / API) · uploading actor · original filename · file hash · initial rights state · attributes and tags applied at upload · duplicate-check signal where one was raised. One-shot · unrecoverable audit

Addition to the original taxonomy. The earlier draft was written entirely around promotion from workflow. Direct upload into the DAM is the other origin path and needs its own provenance event, otherwise every directly-uploaded asset has a blank origin record. Draft-area uploads resolve to this event on commit, not on drop.

Category 3 — Usage and attachment

The workflow ledger. This is the substrate under any "where is this asset being used" question, and under all download reporting.

Event typePayloadTimingRetention
workflow.attachedWorkflow or form reference · attaching actor · attachment point. Marks the beginning of workflow status.Ongoingaudit
workflow.detachedWorkflow reference · actor or system · reason (detached / workflow completed). Marks the end of workflow status.Ongoingaudit
url.mintedRendition template · output format and dimensions · rights state evaluated at mint · requesting actor · expiry · delivery target.Ongoingaudit
asset.downloadedRendition or format delivered · actor · whether part of a bulk download, with batch reference · originating surface.Ongoingaudit
asset.sharedShare type (collection / link) · recipient context · expiry · actor. External recipients recorded as context, not as identified users.Ongoingaudit

url.minted and asset.downloaded are not the same event. A minted URL is authorisation; a download is delivery. They diverge — minted and never fetched, or minted once and fetched repeatedly. Collapsing them makes both numbers wrong. Mint is the rights-enforcement record; download is the usage record.

Category 4 — Access

High volume in aggregate, so the split between compliance signal and engagement noise matters more here than anywhere else.

Event typePayloadTimingRetention
restricted.accessedAsset · actor · rights state at access · surface. Scoped to rights-gated assets only.Ongoingaudit
access.deniedAsset · actor · attempted action · denial reason.Ongoingaudit
permissions.changedAsset or scope · permission set before → after · actor.Ongoingaudit
asset.viewedAsset · actor · surface. Aggregate counter, sampled.Ongoinganalytics

Two things hide inside "view" and they belong in different places. Access to a rights-restricted asset is a compliance signal — low volume, high value, exactly what an audit asks about. A generic view is an engagement metric wearing an audit-event costume. Keeping generic views out of the audit ledger is what keeps the ledger readable.

Category 5 — Product telemetry

Analytics-class from the outset. Same envelope, same store, different retention. Included here so it rides the existing pipe rather than becoming a second instrumentation project.

Retention policy

Decided before events flow, not after. Deciding late means either over-retaining view spam or under-retaining evidence.

ClassContainsRetentionRead paths
audit Provenance, lifecycle, usage/attachment, restricted access, permission changes. Long. Full fidelity, no sampling, no aggregation. Retention period is a governance decision. Asset audit trail · compliance export · provenance lineage.
analytics Generic views, search telemetry, AI tagging telemetry. Short at full fidelity, then rolled up. Sampling permitted. Adoption dashboard · AI quality reporting · search health.

Display scope

Retention class is not the display filter. Retention governs how long an event is kept; display governs whether a user ever sees it. The asset audit trail shows a subset of audit-class events — being audit-class is necessary but not sufficient.

SurfaceShowsNever shows
Asset audit trail
User-facing panel		 asset.promoted · asset.uploaded · rights.changed · lifecycle.changed · approval.changed · attributes.changed · tags.changed · asset.renamed · version.created · workflow.attached · workflow.detached · asset.downloaded · asset.shared Everything in Category 5. Plus url.minted, access.denied, restricted.accessed, permissions.changed, link.severed, asset.viewed.
Admin / compliance view All audit-class events, including url.minted, access.denied, restricted.accessed, permissions.changed, link.severed, asset.deleted. Category 5 in full.
Reporting and dashboards Aggregates over both classes. Never individual audit rows. Identified actors in analytics aggregates.

url.minted is captured but never displayed to users. Mint, purge, revoke, derivative, and edge are infrastructure terms and do not appear in user-facing copy. The event is recorded because it is the rights-enforcement record; it surfaces only in the admin and compliance view.

Deliberate exception to one-event-per-transition: tag acceptance writes two events. tags.changed records the state change — long retention, shown on the panel in plain language. tag.accepted records model quality — confidence, model reference, auto versus user acceptance — rolled up and never displayed.

These answer different questions with different lifespans, and retention_class is a single field, so one event cannot serve both. The alternative is a compliance ledger carrying model version numbers. Do not merge them.

Out of scope

  • The audit trail UI. A read path over this stream, specified and built separately.
  • Dashboard views. The dashboard container and its first views read the asset index, not this stream. An event-backed activity view comes later — the view contract should reserve room for it, but it is not part of capture.
  • Compliance export format. Downstream of retention policy.
  • Role-based display scoping. Which roles see the admin and compliance view is a read-time permission question, deferred. The surface-level split is specified above; the role mapping is not.

Open questions

BlockerWho mints unity_context_id, and when?

The join key has to originate on the Unity side and be passed across the iframe boundary. Same contract question as the breadcrumb postMessage work. Until it is answered, the envelope cannot be finalised and no emitter can be written.

Blocks: event schema. Needs the Unity-side contract confirmed.

BlockerWhen does "on a workflow" begin and end?

The attach and detach events are only meaningful once that boundary is defined, and it is the same question that gates workflow-locking. Without it the usage ledger records transitions whose semantics are undecided.

Blocks: usage and attachment emitters. Outstanding consult.

OpenRetention period for the audit class

Needs a number, driven by client contractual commitments rather than by storage cost. Analytics rollup thresholds follow from it.

Governance decision, with infrastructure input.

OpenField overlap with auto-attribution

Auto-attributed metadata is carried into the asset at promotion and is also listed in the provenance payload. Confirm whether the event records the attribution decision or merely references the attributed values, so the two do not duplicate.

Confirm against the auto-attribution scope before build.

OpenExternal actors on shared links

A guest downloading via an external share link is an external actor with no user record. Decide what identifies them in the payload — share token reference is the conservative answer, but it needs confirming against data-protection posture before it is written into an append-only store.

Needs a governance answer, not an engineering one.

Draft for build. The audit and analytics split, the four-category structure, and the capture-and-display principle are settled. Event type names, payload fields, and the two blockers above are open to change until the schema issue is closed.
