Every active/selected/focus state is cerulean — --primary #0b6e8a for bars and fills, --accent-3 #E4F5FC for tints. Red is never an active state — it's destructive/error only. A red "has-selection" bar anywhere is a bug.

Button tiers — by what the action does, not how it looks

One question per button: what happens to the system when it's clicked? The answer is the tier — not how important it feels. Exactly one Primary per visible context; two filled buttons side by side means one of them isn't Primary.

Template tab, resolved: Create New Template is Primary (new data); Edit Focus Point is Secondary (modifies context). Two fills competing was the bug.

Selectable list-item — hairline-separated, three states

Checkbox + thumbnail + title (14 / 600) + spec (12 / muted). Rows separated by a hairline, never a box each — same call as the filter facets, and it matters more the longer the list. Selected: cerulean tint + 2px left edge + checked box. Unavailable rows stay visible with a red reason (disable, don't hide), never removed. The Template list and any thumbnail picker build from this.

Disclosure vs navigation — triangle vs chevron

Two glyphs, one meaning each. A control that expands content in place uses a filled triangle; a control that takes you to another view uses a chevron. Never the reverse — and both are SVG, never a Unicode text glyph (text triangles render at inconsistent size and weight, which is exactly the bug this replaced).

GlyphMeansWhere
Triangleexpand / collapse in place — rotates: right when closed, down when openfilter facets, edit attributes, collection group headers
Chevronnavigate — drill into another view; static, points the way you gocollection rows (open the collection)
Precedent the users already know: the filled triangle is the macOS / Finder disclosure control (opens in place); the chevron is the iOS row indicator (pushes to a detail screen). A panel may use both — a group header opens with a triangle, its rows navigate with a chevron — and that reads clearly precisely because the two glyphs never swap jobs.

Navigation rail — one component, two densities

The asset panel's left rail is one component shown at two densities, never two different rails. Which density you see follows the context, not the mode: a viewer recedes so the asset is the hero; an editor scaffolds because it's a work surface. Labels are posture, not function — the panel header names the active destination in both modes, so no label is ever needed for wayfinding.

DensityWhereWhy it's this density
Icon-onlyLightbox (view)Asset is the hero; chrome recedes. Tooltip + aria-label carry the name on hover; the header names where you are.
Icon + labelEditFocused work surface carrying tools the viewer doesn't (Adjustments, Template, AI). The label is scaffolding for a task context — not a functional requirement.
Locked across both densities — this is what makes it one component: identical icon vocabulary, order and glyph for every shared destination (Details · Attributes · Rights); identical active and hover states — Scale's .icon-nav-item (active: cerulean fill + edge bar; hover: --surface-muted, dark twin tokenised). Functional groups within a rail (destinations | tools) are divided by a single hairline separator — a category boundary, one per group, never per-item. aria-label is always present in both modes for accessibility, but a visible tooltip appears only where the name isn't already shown: the icon-only viewer, yes; the labelled edit rail, no — a tooltip repeating a visible label is redundant (a labelled item earns one only if its label truncates). Similar is a lightbox-only view destination — discovery has no authoring role in edit — so it sits in the viewer rail after Attributes and never in edit; a view-only destination is legal, an undocumented one is the breach. Download is viewer-rail-only too — in edit it is the top-bar action, not a rail stop. AI is no longer a rail destination at all: it lives inside Tags, under Attributes. File metadata is no longer a rail destination at all: it folds into Details as a read-only, collapsible section, identical in edit and lightbox (SCR-174). Edit-only tools (Adjustments · Template · AI · Versions · Embed) follow the same treatment; a tool keeps a label only when its glyph can't carry meaning alone — fix the glyph first, don't add a label per tab. Rejected: "edit gets labels, read-only doesn't" as a mode rule — mode is not the axis. The shared icon set stays identical so muscle memory transfers between modes; the failure this prevents is the two rails silently drifting into different icons or order.

The three shared destinations (Details · Attributes · Rights) use the same glyph, same order, same active state (cerulean tint + edge bar, on Attributes here) in both. Each mode adds its own outside the shared set: the viewer inserts Similar after Attributes (lightbox-only discovery) and keeps Download as a rail stop; edit instead groups into three bands — describe (Details · Attributes), transform (Adjustments · Template), manage (Versions · Rights · Access) — with Download demoted to the top bar and AI folded into Tags under Attributes. The order is a describe → transform → manage gradient, and roughly descending frequency: identity and catalog first (opened most — Attributes now also carries Tags and AI); the editing tools walled in the centre so edit reads as a workspace, not a menu; lifecycle and governance last (consequential but occasional). Versions groups with Rights and Access because all three concern the asset over time — not its content or its pixels. The viewer drops the labels; edit shows them. Theme (light vs dark) is a separate axis, not the density. The active state is Scale's .icon-nav-item.active — --primary-subtle fill + --primary-default edge on light; on the dark edit rail Scale has no active-tint token, so it uses a tokenised --primary-tint-dark (contrast-checked against stone-800), a gap flagged back to Scale. The shared spine is identical across modes — that's what makes them one component; the per-mode additions are legal precisely because they're documented as view-only or edit-only, not smuggled into the shared set.

The consistency laws

  • One role, one size, one weight — everywhere. A field label is 12 / 400 in the lightbox, the edit panel, and the upload preview. A role can't render two ways across surfaces; if it does, it's drift.
  • Only ladder sizes. 16 / 14 / 12 / 11. There is no 13px, 15px or 18px — reaching for an in-between size is the first symptom of a panel drifting.
  • Dense data is 12px, prose is 14px. Attributes, metadata, options, chips → 12 (font-size-1). Descriptions and reading text → 14 (font-size-2).
  • Hierarchy is weight and colour, not size. Don't introduce a size to make something stand out — change its weight or colour, or remove chrome from what's around it.
  • Cerulean is the only accent. Active, selected, focus, has-selection — all cerulean. Red is destructive only.
  • Inter, nothing else. One typeface across every surface.

Conformance test: does this element use a ladder size and its role's weight from the table above? If yes, it conforms. If it uses 13px, a second size for emphasis, or a red active state — it doesn't, and that's a bug, not a decision.

4 · Cross-cutting rules

  • Casing: header = sentence case · section eyebrow = UPPERCASE · field label = sentence case. (Today: "TIME-BASED RIGHTS", "File Versions", "Attributes & tags" all disagree.)
  • Borders encode grouping/state, not default chrome. Grouped items use the border-collapsed list; no per-item floating pill. Filter/search fields carry no resting border (ring on focus only) — equal-weight borders on parent, search, and every row are what create the "border, border, border" wall and kill hierarchy.
  • Actions & save: field edits stage to the global Save (shell). Structural/immediate actions get their own control + confirm and bypass Save — Upload new version, Transfer ownership, Grant access. Don't let an immediate action masquerade as a staged edit.
  • Light/dark parity + one component across contexts: the library filter rail (light) and the edit panel (dark) are the same Attribute-group component — theme is a token swap, filter-vs-edit is a behaviour swap (filter adds facet counts). Same structure, hierarchy, accent and spacing; don't maintain two. Cerulean is the only selection accent in both.
  • Tool tabs inherit the chrome kit, not the field body. Adjustments, Template and generative AI share the shell, the header, and all shared chrome — section headers, the five button tiers, list-filter search fields, list separation (hairline, never a box per item), disable-don't-hide. Only the unique tool surface is exempt: the canvas (crop / focus point) and the render output. A tool tab is mostly chrome wrapped around one bespoke surface — govern the chrome, exempt only the surface.
  • Bulk is a delta editor, not a form. Every selectable kit component carries a partial state (checkbox → indeterminate, chip → dashed + count). Edits patch the selection (additive / remove), never overwrite. Surfaces that can't operate across a selection — per-asset AI suggestions, single-asset adjustments — go inert + explained, not greyed ambiguously and not hidden.

Shared with the edit contract — must match exactly

  • Same components — parent section → child groups → options, search field, accordion, checkbox. Build the kit once.
  • Same hierarchy treatment — parent is a header (no box), children indent under a neutral nesting rail. Hierarchy from contrast, not borders.
  • Facet groups separated by one hairline + a bold header, never boxes. On a long list grouping must be clear — that is the condition for tolerating density — but a box per group is removable chrome. Bold header anchors, hairline divides, proximity binds. Active groups get a 2px teal edge on the header only + the count pill (SCR-118); a full-height border down an expanded group was rejected as noise on long lists — do not reintroduce it.
  • Cerulean is the only selection accent — header-only, has-selection + count. The rail's current red bar is a deviation to remove.

Filter-specific — what legitimately changes

  • No Save model. Filters apply to the result set live — there's no staged edit, no "unsaved change" pill, no Save button. (Edit stages → Save; filter does not.)
  • Facet counts on every option Host City Poster 40 — how many assets match. Edit mode has none.
  • Zero-count options Wordmark 0 render disabled but present (disable, don't hide), not removed.
  • Saved Filters — save/recall filter sets. A filter-only section with no edit equivalent.
  • Active-filter chips + clear above the results: Official Mark: Host City Poster ✕ Clear all
  • Rail collapse — the header toggle hides the rail to reclaim grid space.
  • Roll-up = active filter count (Attributes 1) — here it counts applied filters, not "facets with a value."
  • No tri-state / bulk. Filtering isn't a delta editor — there's no partial state, no indeterminate. (That's an edit-bulk concept only.)
  • Tags is a filter facet here — same Tags component, but selecting filters the library rather than writing a tag to an asset.
One component, three contexts. The UI team builds the Attribute-group / search / accordion kit once. Filter rail = light + filter mode (this doc). Edit panel = dark + edit mode. The differences above are props — theme, counts on/off, save vs live, tri-state on/off — not separate components. If a difference here isn't on the list, it's a bug, not a decision.

Lightbox-specific — what changes

  • Every component is read-only — Title/Description are static text, attribute values are rows not editors, tags don't show a remove ✕, accordions open but never reveal inputs.
  • View chrome — top bar (prev/next, filename, Add to Queue · Download · ⋯ · close), bottom bar (asset nav, zoom, Esc, "N of M"), and the rail-collapse toggle. None of this exists in edit.
  • No AI Tags tab. AI tags fold into the Tags section with the astroid marker (SCR-120). AI is a property of a tag, not a destination.
  • Attributes + Tags share one tab as two sections (above). No separate near-empty tabs.
  • Similar — read-only result grid, match-% badge with the astroid. Download — the one interactive surface in view mode (version select + CTA). Rights — read-only, with status badges (e.g. Archived) and the auto-archive banner.
  • Empty vs absent — a present-but-empty field shows a muted placeholder ("No description"); a field that doesn't apply to the type is omitted, not shown blank.

Carry-over fixes — already specced, must land here too

  • Details "1 dpi" (image 3) is the unfixed resolution bug from SCR-154 — show DPI only when the file declares a resolution unit, else suppress. The lightbox inherits the fix, it isn't a new decision.
  • The File-metadata section in Details must not leak the temp ingest artifact — tmpxy38n3o5.jpg, /tmp, 100600, MIME/encoding noise. That's SCR-155's allowlist + temp-path fix, and it now governs the section rather than a standalone tab.

Read-only field display — one layout

  • One row, every field: a two-column grid. Label column, value column, value left-aligned. Short values sit on one line; long values (filename, URL, GPS, a long copyright) wrap within the value column — the row structure never changes and the value column keeps a single left edge you scan straight down.
  • No inline-vs-stacked, no per-field classification. There's nothing to know about a field's length in advance — wrapping handles it. The layout can't "jump" within a list because every row is the same row.
  • The editable zone is the only exception — Title and Description render full-width (label above value), because Description is prose and that's how those fields appear as inputs in edit mode. They sit above the facts list, never interleaved into it.
  • Scales by construction. Allowlisted metadata (SCR-155) keeps labels short so the label column stays tight; everything else is just rows. Same grid in Details (including the File-metadata section), edit read-only, and upload preview — a field renders identically everywhere it appears.
  • Rejected: a per-field "inline if short / stacked if long" rule — it makes one list alternate between two row shapes and flips value alignment, so values stop sharing an edge. Right-aligned values were the only thing that ever forced an exception; left-aligned wrapping removes the need.
One component, three contexts — done. Edit (dark · edit) · Filters (light · filter) · Lightbox (light · view). The lightbox adds no new components and no new tabs beyond the rail set Details · Attributes · Similar · Rights · Download — four shared destinations (Details · Attributes · Rights · Download) plus Similar as a lightbox-only view destination. File metadata is no longer a rail tab; it folds into Details as a read-only section (SCR-174). Read-only is a state of the kit, not a separate build.
