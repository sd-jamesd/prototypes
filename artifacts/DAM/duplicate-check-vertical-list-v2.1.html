Duplicate Check on Upload — with resolved states

Builds on v2's vertical list model. Adds resolved-state treatments for the three per-match outcomes, removes the disabled chevron on Ready rows, renames the filter toggle for accuracy, and clarifies the "Use this instead" action with an inline reference to the existing asset.

Supersedes: v2 (vertical list, base model)
Date: 29 May 2026
What's new in v2.1
  • Removed disabled chevron on Ready rows. Chevron now only renders when there's something to expand. Its presence becomes a state indicator.
  • Three resolved states — Skipped, New version, Will upload. Each flagged row collapses into one of these once actioned.
  • "Use this instead" earns its label via an inline reference in the resolved row: "Skipped in favour of [filename] · Open in library →"
  • Toggle renamed from "Show only flagged" to "Show only unresolved" — accurate to the new resolved/unresolved distinction.
  • Header counter shows the breakdown: ready, resolved, and still needing review.

4
Interaction model — resolved-state addendum
What each per-match action does, where the resolved row sits, and how to change a decision.

Click "Use this instead" on a match. Source file is removed from the upload batch. Row collapses to the Skipped resolved state with badge "Skipped" and meta-line "Skipped in favour of [match filename] · Open in library →". Header counter shifts the file from "needs review" to "resolved".

Click "Upload as new version" on a match. Source file stays in the batch, marked to upload as a new version of the existing match. Row collapses to the New version resolved state with badge "New version" and meta-line "Will replace [match filename] · Open in library →". On Complete upload, the file is committed as a new version of the existing asset (depends on versioning bridge — out of scope if not available).

Click "Upload anyway as new asset" (escape). Source file stays in the batch, marked to upload as a standalone new asset. Row collapses to the Will upload resolved state with badge "Will upload". Meta-line shows the file's own details; no reference to any existing asset.

Click the chevron on a resolved row. Re-expands to show the original match options with the previously-chosen action highlighted. User can pick a different action; row re-resolves to the new state. No separate "undo" — just choose again.

Click Remove from batch on any row (resolved or not). File is removed from the batch entirely. Functionally equivalent to "Skipped" but without the "in favour of" reference — the user is rejecting the upload outright, not choosing an existing asset over it.

Complete upload becomes enabled when zero rows are unresolved. Button shows the projected file count (e.g. "Complete upload (8 files)" if 2 were skipped from a batch of 10). On click, the batch commits: ready files upload normally, "New version" files version their referenced asset, "Will upload" files upload as new assets, and "Skipped" files do nothing.

5
Edge cases worth a call with the dev
Spec gaps surfaced by the resolved-state work that need a decision before this lands.
Metadata edit on Skipped files
If the user edited description / attributes on a flagged file before deciding to Skip it, do those edits get discarded? They were intended for an upload that's no longer happening. Probably discarded silently — but flag to the user with a small toast on Skip: "File removed from batch. Metadata edits discarded."
Metadata edit on New version files
If the user edited metadata before choosing "Upload as new version", do those edits overwrite the existing asset's metadata, get appended as version-specific notes, or get discarded? Different versioning systems handle this differently. Needs a dev conversation.
Rights on New version with expired match
When a user uploads as a new version of an expired asset (a legitimate refresh flow), what rights apply to the new version? Probably the rights should be reset and prompted for during the upload step — a new version of an expired asset is essentially a re-licensing event.
"Open in library" target behaviour
Opens in a new tab/window? Opens in the same view (losing the upload context)? Best answer: new tab. The user is mid-upload and shouldn't lose that state to verify a reference.
