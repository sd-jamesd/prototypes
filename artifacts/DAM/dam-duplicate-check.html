Design Spec · Hash-Only Model

Duplicate Check on Upload — hash-only

Hash-based exact-match duplicate detection at upload. Only byte-for-byte identical files are surfaced as duplicates. Near-duplicate discovery (visually similar but not identical) is handled separately via Similarity Search, where users are browsing rather than committing.

Date: 16 Jun 2026
Trade-offs of hash-only
What we gain
  • No false-positive friction for legitimate near-duplicates (product shots with language variants, market localisations, slight crops).
  • Upload flow stays fast — no embedding compute per file at upload.
  • All file types covered, including PDFs, video, Office docs, design files. PSD/AI/TIFF/EPS now work the same as JPG/PNG/WebP.
  • Simpler dev work. Hash matching is engineering-trivial compared to embedding-based detection.
  • No threshold tuning needed at upload. No QA cycle around "is 87% high enough?"
What we lose
  • Near-duplicates (resized, re-encoded, slightly cropped) not caught at upload. User can create unintended duplication.
  • Discovery of these moves to Similarity Search (post-upload, browse-time). Library-level cleanup tooling is out of scope for this release and not yet scoped as a separate feature.
  • Slightly higher long-term duplication burden in libraries — flagged for future cleanup work, not addressed here.
  • PDF/video/design libraries get exact-match dedup only. Worth communicating as a known Phase 1 limitation when explaining the feature.

Annotations
Single deterministic flagged state
Only one flagged state exists: Identical match — byte-for-byte hash equivalence. No "possible" or "maybe" framing, no AI signalling — the check is deterministic. Expansion heading reads: "This exact file already exists in your library".
Multiple hash matches
Expansion can return more than one match when the library has internal duplicates of itself (legacy data, migrations, prior uncaught duplicates). UX handles this correctly. Cleanup of library-level duplicates is out of scope for this release — to be addressed by future tooling, not yet scoped.
Per-match vs file-level scope
Upload as new version is per-match — versioning binds to a specific existing asset, so the button sits in the match card. Use existing and Upload as new asset are file-level decisions — they don't reference a specific match, so they live below the match list.
Expired match — version still allowed
"Upload as new version" stays active on expired matches — it's the re-licensing flow (new content, new rights window, attached to the existing asset record). The EXPIRED badge on the thumbnail signals the state without disabling the action.
Multi-match handled gracefully
When the library has internal duplicates (multiple records with the same hash), the per-match versioning option lets the user pick which one to version. The file-level Use existing and Upload as new asset don't require a pick — they're file-scope decisions.
All file types covered
PDFs, video, Office docs, AI, PSD, EPS, TIFF — all hash-checked the same as JPG/PNG/WebP. No file-type-specific messaging at upload; every format follows the same flow.
Resolved-state model unchanged
All three resolved outcomes share consistent treatment — badge style, meta-line pattern, "Open in library" affordance, and chevron-to-re-open. The trigger differs but the resolution UI is uniform.

Initial check — loading state
Immediately after upload, each file's SHA-256 hash is computed locally and compared against the library. Rows show a checking state until the response returns, then transition to Ready (no match) or Identical match (duplicate found). Per-file, parallel — smaller files finish first.

Annotations
Checking badge
Info-tinted background with spinner icon and the word Checking. Spinner rotates at 0.9s/revolution — fast enough to feel active, slow enough to read as deliberate.
No flagged border yet
Rows in checking state look like Ready rows (no left-border accent). The amber border only appears after the check returns a hash match — never speculative, never preemptive.
Per-file, parallel
Each file's check runs independently. Smaller files (a 312 KB PNG) finish in well under a second; larger files (17 MB JPG) can take 2–3s for the hash compute plus the server lookup.
Header status updates live
10 files · checking 7 · 3 done — counts shift as each file resolves. Once all checks complete, the header swaps to the resolved-state breakdown shown in Section 1.
Remove still available
User can remove files from the batch during checking. Cancels the in-flight check for that file. No data sent to server beyond what was already in flight.
Complete during checking
Complete upload stays enabled. If clicked while files are still checking, a confirmation appears: "5 files are still being checked. Proceed without verification, or wait?". User chooses — system never blocks.
"Taking longer than usual"
After ~15s in checking state, the meta line acknowledges the wait inline: "…taking longer than usual". Same badge, same spinner — just an honest copy change. Stops the user wondering if something's broken without prematurely declaring failure.
Check failed + Retry
After timeout or server error, the row resolves to Check failed with a muted error badge. Meta line: "Couldn't verify against library". Retry icon (refresh) sits in the chevron slot — single click puts the row back into Checking state.
Timeout calibration
Per-file timeout scales with size: ~10s for small files (<5 MB), ~30s for mid (5–50 MB), ~60s for large (50 MB+). Dev-side configuration, not user-facing. Hash compute is local but server lookup is the network round trip.
Complete with failed rows
If the user wants to ship without resolving a failed row, Complete upload prompts a confirmation: "3 files couldn't be checked — proceed anyway?". Better than blocking — server hiccups shouldn't take the user hostage.

5
Interaction model
Row + expansion + resolved-state model. Flagged rows are always Identical match — single deterministic flag state derived from SHA-256 hash comparison.

Click "Use existing" below the match list. File-level action — doesn't bind to any specific match. Source file is removed from the upload batch. Row collapses to the resolved state with badge "Using existing" and meta-line referencing the existing asset (or summary count if multiple matches).

Click "Upload as new version" on a specific match. Per-match action — binds the source file to that match as its new version. Source stays in the batch, marked to commit as a new version of the referenced match on Complete. For hash matches this means a new version event with identical file content; useful for audit/compliance flows or re-licensing of an expired match.

Click "Upload as new asset" below the match list. File-level action — source commits as a standalone new asset. The duplicate-check signal is logged against the asset record; surfacing/cleanup of library-level duplicates is out of scope for this release.

Click the chevron on a resolved row. Re-expands to show the original match options with the previously-chosen action highlighted. User can pick differently. No separate undo.

Complete upload stays enabled throughout the session. The duplicate check is a flag, not a gate — the user's right to upload is honoured at every step. On click, the system handles each row according to its state:

  • Ready → commits as new asset
  • Resolved as Using existing → discarded (file not uploaded)
  • Resolved as New version → commits as new version of the referenced match
  • Resolved as New asset → commits as new asset, with duplicate-check signal logged against the record
  • Unresolved (Identical match) → defaults to uploading as new asset, same as if the user had chosen that resolution explicitly
  • Checking → upload waits for check to complete, OR proceeds without verification if user confirms
  • Check failed → upload proceeds if user confirms; signal logged that verification did not complete

Confirmation dialog appears when Complete is clicked with any unresolved, still-checking, or failed rows in the batch. Dialog lists the breakdown and asks for confirmation:

Continue with upload?
Your batch contains items that haven't finished resolving. These will commit with the defaults shown below.

The dialog only appears when at least one row is in a non-clean state. If all rows are Ready or cleanly resolved, Complete fires immediately with no interruption. Item counts shown in the dialog reflect the actual state of the batch — sections only appear if they contain items (a batch with only unresolved duplicates won't show the checking or failed lines).

6
Edge cases worth a call with the dev
Edge cases the hash-only model surfaces or simplifies. Library-level duplicates (where the library itself already contains internal duplicates) are newly visible at upload.
Library-level duplicates (multiple hash matches)
When the library already contains internal duplicates (same hash, separate asset records — typical legacy/migration artifact), the upload check returns more than one match. UX handles this correctly (just renders more match cards). The existence of these library-level duplicates is a separate problem; surfacing and cleanup is out of scope for this release and not yet scoped as a follow-on feature.
Metadata edit when Use existing is chosen
If the user edited description / attributes before clicking Use existing, those edits get discarded. Flag with a small toast on resolve: "File removed from batch. Metadata edits discarded."
Metadata edit on New version files
For hash matches, the file content is identical to the existing asset — so what does "Upload as new version" actually change? Usually metadata or rights or audit context. Behaviour of edits applied at upload (overwrite existing metadata, append version-scoped notes, or discard) needs a dev call.
Rights on New version with expired match
Versioning an expired asset is a legitimate re-licensing flow. New version should reset rights to the uploader's input rather than inheriting the expired state. Worth being explicit in the spec.
"Open in library" target behaviour
Opens in a new tab. The user is mid-upload and shouldn't lose that state to verify an existing reference.
7
Multi-select and bulk edit
Checkboxes on every row let the user select multiple files and apply metadata, attributes, and rights in bulk. Selection state coexists with duplicate-check state — the left-border accent (system signal) sits on top of the selection tint (user signal). Both surfaces work without conflict.

Annotations
Checkbox column
Sits at the leftmost position of every row, always visible. 18×18 box with cerulean fill on checked. Click target extends across the column for ease of selection.
Selection coexists with flagged state
Two-channel feedback: left-border accent for system state (flagged for duplicate), background tint for user state (selected). They occupy different visual layers — a flagged + selected row shows amber on its edge, cerulean tint on its body.
Selection works on every row
Checkboxes stay enabled across all states — including Using existing and Check failed. Selection is for batch operations; Remove from batch works on any selection regardless of state. Metadata edit applicability is handled in the panel, not by blocking selection at the checkbox.
Bulk action bar replaces toolbar
When 1+ rows are selected, the normal "10 files · 42.7 MB total" toolbar swaps to "3 files selected" with Clear selection link and Remove from batch action. Clears back to default toolbar when selection drops to zero.
Side panel shifts to multi-mode
Filename area shows "3 selected files" instead of an individual filename. Description placeholder updates to "Add a description for all 3 files…". Helper text confirms: "Applying standard tags to 3 files."
Mixed-state selection
When selection includes a Using existing or Check failed row alongside editable rows, the panel header shows "Editing metadata for 2 of 3 selected files" and a helper line confirms "1 file (Using existing) won't receive these edits". On Complete, edits apply only to the editable rows — the others stay as they are.
Standard selection patterns
Click checkbox to toggle. Shift+click for range. Cmd/Ctrl+click for additive. Space toggles focused row. Header checkbox (not shown here) would select-all with indeterminate state when partial.
Tri-state attributes on multi-select
Same pattern as library bulk edit: attribute checkboxes show empty (not set), checked (set on all), or tri-state with N of M count (set on some). At upload, most files start with empty attributes — so the default state is empty across the board. Tri-state only appears in the edge case where the user has already edited some files individually before opening multi-select.
