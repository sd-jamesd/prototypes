1. What is rights management?

Rights management controls who can download what, and in what quality. It exists to prevent assets from being used outside of their licensing terms — wrong country, wrong audience, wrong time window, wrong channel.

Every rights-managed asset in the DAM has metadata defining where, when, and by whom it can be used. When a user tries to download an asset, the system evaluates their profile against the asset's rights metadata and returns one of four outcomes:

FULL

All download options available.

RESTRICTED

Preview + Watermarked only. Original disabled.

NOMATCH

No download. May see Request Access.

BLOCKED

Time-based block. No download at all.

Assets that are not rights-managed (no licensing metadata set) skip all of this — any user with basic DAM access gets full download.

Key principle: Rights affect download quality, not visibility. Users can always see and preview a rights-managed asset in the DAM. They just can't always download the original.

2. The decision tree

When a user requests a download, the system evaluates access in strict priority order. The first matching condition determines the outcome — everything below it is skipped.

1
File NOT rights-managed?
If the asset has no rights metadata (UsageChannels is empty), skip everything → FULL
2
Superadmin?
JWT claim superadmin = true bypasses all rights logic → FULL
3
Approved via sd$fileAccess?
If an approved record exists for this user + file in sd$fileAccess → FULL (but time blocks still apply — see step 4)
4
Time-based block?
CDN blocked, UserAccess soft, expiry, embargo, usage window — checked in priority order. First match → BLOCKED
5
EditComm = "editorial"?
Editorial assets are restricted regardless of dimensional match → RESTRICTED
6
RequireExplicitApproval = true?
Asset requires owner approval before download. Pending and rejected records are ignored (only approved counts, and that's checked at step 3) → NOMATCH + Request Access button
7
No user profile?
User has no row in sd$standardUserData → NOMATCH
8
No dimensions defined on file?
File has null territory, audiences, and channels — no criteria to score against → NOMATCH
9
Dimensional scoring
Compare user profile (territory, audiences, channels) against file metadata. Result depends on how many dimensions pass — see Section 4.
Order matters. A superadmin (step 2) bypasses everything including time blocks. But an sd$fileAccess approved user (step 3) is still blocked by time checks (step 4). This is intentional — time blocks are legal constraints, not permission issues.

4. Dimensional scoring

When the decision tree reaches step 9, the system compares the user's profile against the file's rights metadata across three dimensions: territory, audience, and channel.

Each dimension is evaluated independently — either the user's value intersects with the file's allowed values (pass), or it doesn't (fail). The file may define all three dimensions, or only one or two. Only defined dimensions are scored.

Scoring rules

Condition Result
All defined dimensions pass, none fail FULL
At least one passes, at least one fails RESTRICTED
Zero dimensions pass NOMATCH

Partial dimensions

When a file only defines some dimensions (e.g. Territory only, no Audiences or Channels), the rule still applies but over fewer dimensions.

Dimensions definedUser matchesResult
Territory only (1/3)Territory ✓ (1/1)FULL
Territory + Audiences (2/3)Territory ✓, Audiences ✗ (1/2)RESTRICTED

Empty profile gotcha: If a user has a profile row but all fields are blank (U16 in the test plan), they fail every dimension — this is NOMATCH, not an error state. QA should verify this user sees the same UI as a zero-match user.

What each dimension represents

Territory

Where the asset may be used geographically. The user's assigned territory (from their profile in sd$standardUserData) is compared against the file's Territory array. Matching is case-insensitive. A user may have multiple territories (e.g., "USA, Canada").

Audience

Who the asset is intended for. Defines the target audience group — Marketing, Editorial, Broadcast, Events, Design, Agency, etc. This is a usage constraint, not a role. A "Marketing" audience means the asset is licensed for marketing use, not that only marketers can see it.

Channel

How the asset may be distributed. Digital, Print, Social, Broadcast, OOH, etc. Since the DAM has no syndication tools, channel is enforced at download as an access gate — the system cannot control what happens post-download, but it can restrict who gets the original file.

5. Time-based blocks

Time blocks are checked before dimensional scoring (step 4 in the decision tree). They represent legal or contractual time constraints. Even a user with full dimensional match gets blocked if a time condition is active.

Critical: Time blocks override sd$fileAccess approvals. Even U13 (Broadcaster Partner) with an approved record cannot bypass an embargo. The only exception is superadmin (step 2 fires before step 4).

Time blocks are evaluated in priority order — the first match returns a 403 and the rest are skipped:

Embargo vs Expiry — the difference

Embargoed

The asset is not yet allowed to be used. Campaign not live, press release under embargo, sponsorship imagery restricted until a launch date.

The embargo lifts automatically when the date passes. No manual action needed — the system re-evaluates on next download attempt.

Expired

The asset is no longer allowed to be used. License ended, model release expired, campaign period closed.

Expired assets remain visible in search (users still need them for reference and to avoid re-uploading). Full download is hard-blocked.

AutoArchive

When AutoArchive = true and the expiration date is past, the file is removed from search results for non-admin users. Superadmins can still see and download it. This is a separate visibility control, not a rights state — it fires before the download modal is even reachable.

6. Approval workflow — Request Access

Some assets require explicit approval before any download is permitted. This is set per-file via RequireExplicitApproval = true.

How it works

When a user encounters a file with RequireExplicitApproval, the decision tree reaches step 6. At this point, the system checks sd$fileAccess for an approved record matching this user + file combination. Only approved status counts — pending and rejected records are ignored.

sd$fileAccess statusEffectWhat user sees
No record existsNOMATCHRequest Access button
pendingIgnored → NOMATCHRequest Access button (pending doesn't grant access)
rejectedIgnored → NOMATCHRequest Access button (user can re-request)
approvedBypass → proceeds to step 4 (time blocks)Full download options (unless time-blocked)
Approved + time block: If U13 has an approved sd$fileAccess record for F15, but F15 also has an active embargo, the time block (step 4) wins and the user sees BLOCKED. Approval bypasses dimensional scoring (step 9) but not time-based legal constraints.

7. Superadmin bypass

Superadmin is a JWT claim, not a database role. If the superadmin flag is present in the authentication token, all rights logic is bypassed — including time blocks, dimensional scoring, and approval requirements.

QA note: When testing superadmin scenarios, verify the UI shows a superadmin bypass indicator. This prevents admins from accidentally downloading restricted files thinking they're unrestricted. The indicator should be visible in the download modal and on the asset detail page banner.

8. License warnings

Some assets carry license metadata that generates a warning but does not block download. These are informational — the user is told to verify their intended use, but the download proceeds normally.

ConditionWarning message
ownershipSource = "Stock Provider" or "Agency"Stock or agency licensed asset — verify usage rights before downloading.
licenseType = "Third-party Stock"Same as above
licenseType = "Rights-Managed"Restricted licence — confirm permitted use before downloading.
licenseType = "Editorial Only"Editorial use only — commercial use not permitted.
licenseType = "Commercial" or nullNo warning

Warnings appear as an amber banner inside the download modal. They do not affect the access level or disable any options. The user can acknowledge and proceed.

Warning ≠ block. If QA sees a license warning preventing download or disabling options, that's a bug. Warnings are passive and informational only.

11. Glossary

TermDefinition
Rights-managedAn asset with licensing metadata (UsageChannels is non-empty). Triggers the full decision tree evaluation at download time.
Dimensional scoringThe comparison of user profile values (territory, audience, channel) against file rights metadata. Produces FULL, RESTRICTED, or NOMATCH.
sd$standardUserDataThe database table storing each user's rights profile: territory, audiences, channels. Used for dimensional scoring.
sd$fileAccessThe database table storing explicit per-user, per-file access records. Status can be approved, pending, or rejected. Only approved records grant access.
RequireExplicitApprovalA per-file flag. When true, the asset cannot be downloaded unless an approved sd$fileAccess record exists for the requesting user.
EditCommEditorial commercial flag. When set to "editorial", the asset is restricted to low-res/watermark regardless of dimensional match.
AutoArchiveWhen true and the asset is expired, it's removed from search results for non-admin users. Superadmins can still see it.
Time blockAny of the 7 time-based conditions (CDN, UserAccess, expiry, embargo, usage start, window start, window end) that block download before dimensional scoring is reached.
LCD rule (bulk)Lowest Common Denominator. In bulk downloads, only options valid for ALL selected files are shown. If one file restricts Original, Original is removed from the bulk modal.
SuperadminA JWT claim (not a database role) that bypasses all rights logic including time blocks.
PreviewUser-facing label for the low-resolution download variant. 1200px, 75% compression, no watermark. Backend key: lowres.
Watermarked versionUser-facing label for the watermarked download variant. 2000px, 85% compression, diagonal watermark. Backend key: lowres-watermark.
Original fileUser-facing label for the full-resolution master. No compression, no watermark. Backend key: original.
