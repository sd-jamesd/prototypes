Layout B.2 — Title carries search state

Title-as-subject pattern (Scale DAM tokens)

Six states stacked, same skeleton. The page title's data binding is the entire mechanism: when there's no search, it's the library name; when text search is active, it's FIFA WC 2026 › Results for "<query>"; when similarity search is active, it's FIFA WC 2026 › Visually similar to <ref asset>. The library always sits as a quieter breadcrumb segment before the H1. No banners, no tints. Built strictly against Scale tokens — stone for chrome, teal reserved for primary action, active state, and checked filters.

Header layout: two-cluster pattern. Left cluster holds the title block and search element (chip or input) — grows rightward, absorbs all width variation from the title's state change. Right cluster holds result count, view toggle, Queue, Upload — stays right-anchored, fixed X-coordinates across every state. The two clusters never move relative to their respective edges; a flexible spacer between them absorbs the difference.

View-controls row: a single conditional row between header and workspace. Carries applied filter pills, Save action, and result meta. Renders only when filters are applied. The similarity threshold lives in the title block as a popover trigger (not in this row), keeping the row's job consistent: chrome that displays applied refinements to a result set. Library / text search → no controls row when no filters. Similarity → no controls row when no filters. Filters applied → controls row appears, same shape regardless of whether the search is text or similarity. Two-row maximum across every state (header + controls row); single-row baseline.

State 1 of 6

Library — no search, no filters

Baseline. Page title = library name. Search input full width. Pill row absent (conditional). Workspace shows full library.

What's signalling search state
Title slot composition: single line, baseline-aligned. Library segment (13px / 500, text-muted) precedes a separator chevron, then the H1 "Results for "image"" (18px / 600, text-default). Hierarchy through size + colour weight, not stacking. Header height drops by ~12px compared to the stacked version, removing the vertical whitespace the right cluster was sitting in.
Library segment is a link: clicking it returns to the library view. The chip's ✕ does the same thing. Two affordances for the same action because they suit different mental models — one for "exit search mode," one for "navigate back."
Search chip: replaces the full input. Compact. Click to expand and edit, ✕ clears search. Reclaims ~280px of header width that the full input was consuming.
No banner, no tint, no badge. The title swap is sufficient. Per rule #2, teal would be inappropriate here — search state isn't an "action" or "active selection," it's a page-subject change. Stone hierarchy carries the load.

How Scale rules are applied here
Active-group indicator (rule #6): Rights state and File type group headers show a 2px teal accent on the left + subtle count pill in the header. Group children are NOT background-washed — the checked checkbox carries the per-option state.
Applied-filter chip (rule #7): stone-3 background, stone-12 text, ✕ icon stone-muted by default and teal on hover. Two chips don't compete with the title or any primary action.
Count pill — subtle variant (rule #5): the "1" in each active group header. Primary-subtle background, primary-11 text. Count is context here, not action.
No teal anywhere it doesn't earn: applied label is stone, "Clear all" is stone-muted (hover deepens to default text), "Save as filter" is stone-muted (rule #2 — actions in chrome are stone, only the primary action and active state earn teal).
Result meta: "14 of 31 results shown" tells the user how filtered they are within the search. Stone text, just enough emphasis on the count via tnum + bold.

Why this matters
The title still does its job: even with zero results, the page still says what was searched for. Users can read the page identity correctly without needing to look at the empty grid and infer why it's empty.
Two recovery actions, both secondary: "Return to library" (back to no-search) and "Edit search" (refocuses the chip into edit mode). No primary action in the zero state — neither path is more correct than the other, depends on user intent.
Filter panel stays available: power users can apply filters to a zero-result search to refine the no-results query. Edge but worth supporting.

The reference asset also appears as the first card in the grid (marked, no score) so users have an immediate visual yardstick against which to read each match score. What this state commits to
Page identity is "Visually similar to <ref asset>". Not a quoted text query — a thumbnail + filename. The reference filename in a stone-3 chip with mono font reads as data, not chrome. The 32px reference thumbnail inline gives a quick visual anchor.
The reference asset is also the first card. Marked "Reference" with a teal label, bordered teal (rule #6 — teal earns its place as active state). No similarity score on this card because it's not a match — it's the basis for matching. This gives users an immediate "this is what I asked for, and here's how close each match comes."
Threshold lives in the title block as a popover trigger. The threshold is a parameter of the query itself (alongside the reference asset and library scope), not a refinement of the result set. Sitting it inline with the ref-name keeps these query-defining things together. Stone trigger styling — clickable but not announced as an action or active-state. Click opens the popover with slider, presets, and helper text. Closes on outside click or after a value commits.
No controls row in this state. With threshold in the title and no filters applied, the row has nothing to carry. Header is the only chrome above the workspace. State 5b shows what happens when filters get added — controls row appears, but only for the pills.
Similarity score badges (per rule #2): solid teal background with white text on the thumb. Score IS the point here — the count of similarity. This is one of the few legitimate places to use solid teal as a content overlay rather than chrome.
Sort default is "Similarity", not date. Reflects what users came for. Sort is still a stone control — sort selection isn't an "action," it's a property of the view.
Filter panel still works. A user can apply "File type: SVG" to narrow similarity results to just vectors. Filters compose with similarity as they would with text search.
Implementation notes for the dev
Entry point. Similarity is triggered from an asset detail page's "Find similar" action. URL pattern: /library/<id>/similar?ref=<asset-id>&threshold=70. The reference asset id and threshold are query params; this lets users deep-link a specific similarity search and lets the back button work properly.
Reference card behaviour. Clicking the reference card returns to its asset detail page (where the search started). Standard chevron-back interaction.
Threshold debounce. Slider should debounce ~300ms before re-running the similarity query. Otherwise dragging causes a query storm.
Result count semantics. "12 matches" not "12 results." Subtle but matters — matches has the right connotation for similarity (degree-based) where results is right for text search (binary).

What the merged row earns
Controls row matches State 3's shape. Pills cluster on the left, Save + meta on the right. No threshold, no divider — those were the things that made the row feel structurally different from a normal applied-filter row. Now it's the same row component across all "filters applied" states regardless of search type.
Threshold is reachable from the title block. Closed by default. Click to open the popover (see State 5 above for the open state). Threshold value displayed in the trigger — users always know what the current threshold is without opening it.
Sort indicator moved to the header right cluster. Sort is a view control like view-toggle; lives with its functional family. Stone border, stone treatment. The right cluster gains one element but stays compact.
Two rows of chrome maximum, baseline of one. Header always present; controls row appears only when filters apply. Library / text search with no filters = one row. Similarity with no filters = one row. Filters applied (either search mode) = two rows. Same component in every case.
Summary

What the title-as-subject pattern commits to

The page title is the single primary signal that distinguishes a library view from a search results view. When a search is active, the title slot reorganises into a breadcrumb-style trail: library name as a quieter segment (13px, text-muted), separator chevron, prominent H1 carrying the query. All inline at the same baseline — no vertical stacking. The user's eye lands on the H1 first (largest, darkest), reads the breadcrumb context naturally to its left, and parses the page identity in one scan.

Hierarchy is carried through typography (size + colour weight), not layout (stacking). This is the more honest treatment: the library segment is genuinely subordinate to the query, and inline-with-smaller-quieter-text says that more truthfully than "small label above big title."

One thing worth re-flagging from earlier: when the iframe-to-Unity breadcrumb sync ships and Unity shows the library name in its breadcrumb (Home / Digital Archive / FIFA World Cup 2026), the library segment + chevron can be dropped from the DAM page header. The title slot becomes a single H1 — just "Results for "image"". The data-binding accommodates that without any structural changes — same flex container, fewer children.
