Collections panel — GO-LIVE v1

Three visibility states. Scale v4 · Inter · Lucide.

1 · Group by ownership, not visibility

Pinned system collection (Favourites) → My Collections → Shared with me. This is the axis users navigate by and the axis the rights model gates on (§7: creating shared collections is role-restricted; personal isn't).

2 · Left slot = identity. Right slot = status.

The left column is reserved for kind — what the row fundamentally is. Only genuinely distinct kinds populate it (Favourites = system collection). User collections are all the same kind, so their left slot stays empty but reserved, holding a clean text column.

Visibility is mutable state, not kind — so it sits right, smaller and muted, as a status glyph before the count. Flip a collection's visibility and the glyph swaps; the row keeps its place. This matches how the product already renders status (asset rights states sit right too).

3 · Visibility states — go-live (three)

Private — personal collection
Shared internally — named accounts inside the org
Shared externally — verified guests (OTP / password)

Globe = external (it's left the org) — the trust boundary you scan for. The gate mechanism (password vs OTP) is never in the list; that's a detail-panel fact. Public / open (amber globe) is roadmap — it lands with the rights-cleared gallery (SCR-129), not at go-live.

4 · The bookmark is gone from the rows

It stays on the rail as the destination icon; it no longer echoes down every row. Row = name + status glyph + subtle count pill + chevron. 0-count renders as a quiet 0 pill.

⚑ Go-live scope Three states ship now: private / internal / external-verified. Rights are enforced per-asset at the delivery endpoint (§8 row 83 "Based on rights"), which cascades into collections and shares — collections carry no rights gate of their own. Public / open is deferred to the rights-cleared gallery + CDN-mint work (SCR-129); the amber-globe state is designed and held in the roadmap mock.
