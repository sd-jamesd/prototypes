01 — THE CADENCE

Two-week cycles, four-week releases

Dev runs one cycle ahead of QA so neither stream ever waits on the other. Every four weeks a new version ships to the client. v2.0 lands on 11 Sep with a one-week finalisation buffer.

Dev build cycle QA + PM cycle Release shipped v2.0 launch Finalisation only
02 — WHY IT WORKS

Three rules hold the model together

Follow them and the cadence runs itself. Break them and it collapses inside two cycles.

Rule 01
Dev runs one cycle ahead of QA

Dev finishes v1.1 in Cycle 1 and moves straight to v1.2 in Cycle 2. QA picks up v1.1 in Cycle 2 with the whole cycle to test it. Neither stream waits on the other.

The pipeline
Rule 02
The in-flight release always beats the next one

If QA finds a bug in v1.1 during Cycle 2, fixing it outranks v1.2 build. The release with a ship date wins. Without this rule v1.1 rots while v1.2 races ahead.

The bug rule
Rule 03
The buffer week is finalisation, not build

The final week before v2.0 launch is bug fixes and QA only. No new features cross into the buffer. Anything not feature-complete by 5 Sep is v2.1, not squeezed in.

The buffer rule
