Scale Design System

Version 1.0 · Screendragon · Internal Documentation

What Scale Is

Scale is Screendragon's design system. It defines the components, tokens, and patterns used across all product surfaces — work management, digital asset management, and future modules — to deliver a consistent, coherent experience at enterprise scale.

The name comes from the interlocking plates of a dragon's armour: modular, tight-fitting pieces that form a seamless surface. Every component is self-contained and connects to every other component through shared tokens.

Scale covers one product — Screendragon — delivering a consistent experience across all surfaces. The system is token-driven: every visual property is defined as a named value, creating a single source of truth for both design and development.

The documentation targets React (the migration target). The existing Vue codebase should converge on the same token values and visual specs. This documentation assumes you know React, CSS custom properties, and Radix UI primitives.

Token Architecture

Scale uses a three-layer token architecture. This is the foundation everything else is built on.

Layer 1: Primitive Tokens

Raw values — the base palette, spacing scale, type scale, radius values. No opinion about context or usage. Never referenced directly in component code.

Primitive tokens — named values only
--accent-9 #0B6E8A
--neutral-12 #261E1B
--accent-3 #E4F5FC
--accent-9: #0B6E8A; /* Cerulean — step 9 */ --neutral-12: #261E1B; /* Warm neutral — step 12 */ --accent-3: #E4F5FC; /* Cerulean — step 3 */ --spacing-2: 8px; --radius-2: 4px;

Layer 2: Semantic Tokens

Contextual aliases that map primitives to a purpose. Declares what something means, not what colour it happens to be. This is the default reference layer for all component styles.

Semantic tokens — purpose-driven aliases
--surface-default
--surface-subtle
--primary-default
--text-default
--surface-default: var(--color-stone-50); --accent: var(--color-blue-500); --text-default: var(--color-stone-900); --status-active-bg: var(--color-green-100);

Layer 3: Component Tokens

Scoped tokens for specific components where the semantic layer is too broad. These reference semantic tokens, creating the cascade.

Component tokens — scoped overrides
← uses --btn-primary-bg → --accent → --color-blue-500
--btn-primary-bg: var(--primary-default); --tab-active-indicator: var(--primary-default); --bulk-bar-bg: var(--primary-default);
Rule: If you are writing a hex value, an rgb() value, or a pixel value directly in a component's styles, you are doing it wrong. Use a token.

Composition vs. Net-New

When a new feature needs a UI element that doesn't exist in Scale:

The rule: If composing existing primitives would compromise the interaction's UX, design net-new — constrained by the token layer. Otherwise, compose.

A confirmation dialog composed from Modal + Button + Text is a composition that works. A complex multi-step asset upload wizard forced into a Modal composition does not — it needs its own pattern, built with Scale's tokens but not constrained to existing component shapes.

Net-new components must use Scale's semantic tokens for all visual properties, pass Scale's accessibility requirements, and go through the contribution process before entering the system.

Compositions use components as documented and do not need the contribution process — they are applications of the system, not extensions of it.

Colour Strategy

Scale's colour decisions are grounded in competitive analysis and semantic logic, not aesthetic preference. This section documents the rationale behind the three defining colour choices in the system: Cerulean (the teal action colour), the warm neutral scale, and the separation of brand green from functional colour.

Why Cerulean — The Action Colour Decision

Blue owns the action layer in enterprise work management. Across eight direct competitors analysed, seven use blue as their in-product CTA colour — regardless of their brand hue. Asana (coral brand, blue CTAs), monday.com (purple brand, blue CTAs), Wrike (green brand, blue CTAs), and Adobe Workfront (red brand, blue CTAs) all deliberately separate brand identity from functional action colour. Only when the brand is blue (Atlassian, Smartsheet) does CTA equal brand.

Screendragon's brand green (#006B5C) passes WCAG AA at 6.45:1 — technically safe as a button colour. But using green as the primary action colour creates semantic collision with universal "success/on-track" patterns that enterprise users carry across products. Every competitor uses green to mean "done," "active," or "on track." Making green also mean "click here" creates an unnecessary cognitive tax.

The decision: Cerulean (#0B6E8A) — a deep teal-blue at ~195° hue. It sits precisely between Screendragon's green identity (~160°) and the standard SaaS blue (~220°), occupying a hue space no competitor claims. It reads unambiguously as an action colour while carrying enough teal warmth to feel connected to the brand's green DNA.

Teal hue positioning — competitive landscape

The Green Question — Where Brand Colour Lives

The question isn't whether Screendragon keeps its green — it's where green lives. Brand green (#006B5C) dominates the personality layer of the product while the action layer runs on Cerulean (#0B6E8A). This mirrors the Asana playbook (coral for personality, blue for action) and the Wrike approach (green for brand, blue for interaction).

✓ Brand green appears here
Sidebar header accent strip, empty state illustrations, loading indicators, onboarding flows, marketing pages, login screens. These are the places users don't click — where green builds recognition without semantic confusion.
✗ Brand green stays away from
Primary CTA buttons, text links, focus rings, toggle/checkbox active states, selected tabs. These belong to Cerulean (#0B6E8A). Success states use the separate semantic green, not brand green.

Why Warm Neutrals — The Stone Scale

Most enterprise SaaS products use cool blue-grays. Atlassian's deepest neutral is #091E42 (blue undertone). Smartsheet uses #00184D (navy). Notion uses #37352F (one of the few warm exceptions). The cool-gray monoculture makes enterprise products feel interchangeable.

Screendragon's neutral scale is deliberately warm — rooted in SD Dark Charcoal (#261E1B) with a brown undertone that runs consistently through all 12 steps. This creates a subtly friendlier, more human feel without sacrificing professionalism. It's a genuine visual differentiator: warm neutrals are the first thing that signals "this isn't another blue-and-gray SaaS tool."

Warm vs cool neutrals — the rationale

The Recognisability Formula

Screendragon's visual signature is built from three elements working together: warm neutrals (brown-gray, unique among competitors) + brand green sidebar accent + Cerulean action colour = a product that is recognisably Screendragon within 500ms of loading. This is analogous to how Asana's dark sidebar + coral celebrations + blue actions = instantly "Asana."

Three-layer colour architecture
Warm Neutrals
The canvas — surfaces, text, borders. Sets Screendragon apart from cool-gray competitors.
Cerulean
The action colour — buttons, links, focus rings. Named for its position between green and blue at ~195° hue.
Brand Green
The personality layer — sidebar accents, illustrations, empty states. Recognition without function.

Core Controls

3.1 Button — Hierarchy & Action Classification

Every button in Screendragon is classified by what it does to the system state, not by how important it looks. This five-tier system maps action functionality to visual weight, ensuring consistent hierarchy across every page.

Overview

Screendragon surfaces dozens of actions per page — create, filter, export, edit, delete, share, assign, archive. Without a rule-based hierarchy, button styling drifts toward visual sameness (the screenshot problem: four actions, three looking identical) or arbitrary promotion (making "Export" as loud as "Create").

This system classifies every action by what it does to the system state, then maps that classification to a visual tier. The tiers are cumulative — a page toolbar might use Tiers 1, 2, and 3 simultaneously, each at its correct visual weight.

Five tiers at a glance — medium size (40px)
Tier 1 · Creates data
Tier 2 · Reshapes view
Tier 3 · Peripheral utility
Tier 4 · Inline / contextual
Tier 5 · Irreversible removal

Decision Framework

For any new button, ask one question: "What happens to the system when the user clicks this?" The answer maps directly to a tier.

Does clicking create new data or advance the user's core workflow?
Create New, Save, Submit, Publish, Assign, Confirm
Tier 1 · Primary
Does clicking change what the user sees without committing new data?
Filter, Sort, Toggle View, Show/Hide Panel, Edit, Share, Apply
Tier 2 · Secondary
Is the action useful but peripheral — needed once per session, not once per task?
Excel Export, Download PDF, Reset Filters, Expand All, Print, Audit Log
Tier 3 · Tertiary
Does the action target a specific row, card, or inline item?
Row kebab menu (⋯), edit pencil, row delete, drag handle, expand chevron
Tier 4 · Ghost / Icon
Would a mistaken click cause irreversible data loss?
Delete permanently, Remove user, Revoke access
Tier 5 · Destructive
Key Principle
If an action is reversible (archive, hide, deactivate), it uses standard Secondary styling — not Destructive. Destructive is reserved for actions where recovery is non-trivial or impossible.

Distinguish Tertiary from Ghost
Tertiary buttons use accent colour text — they're still branded, still recognisable as clickable. Ghost buttons use neutral text and are intentionally de-emphasised. If the action is peripheral and page-level, it's Tertiary. If it's contextual and inline on an item, it's Ghost.

Rules & Constraints

The One-Primary Rule

Every visible context (page toolbar, modal footer, side panel) gets exactly one Primary button. If you're placing two filled Cerulean buttons side by side, demote one to Secondary. The most common mistake is promoting "Save" and "Publish" both to Primary — one of them (usually "Save as Draft") should be Secondary.

✓ Do
One Primary forward action, one Secondary alternative, one Ghost escape.
✗ Don't
Two Primaries compete. The user can't quickly identify the forward action.

Toolbar Reading Order

Primary goes to the far right (highest visual weight, last in reading order). Tertiary goes to the far left (lowest weight, first in line). Secondaries sit in between. This creates a natural left-to-right hierarchy from least important to most important.

✓ Do
Tertiary → Secondary → [spacer] → Primary. Natural weight progression.
✗ Don't
Primary on the left loses its position as the destination action.

Destructive Always Confirms

No destructive action executes in a single click. Every Tier 5 button opens a confirmation dialog. The dialog's footer uses Ghost Cancel + Destructive Primary Confirm — never Secondary Cancel (that would visually compete).

Don't Promote Peripheral Actions

Export, Print, Download, and Settings are always Tertiary. Giving them Secondary treatment (visible border) makes them compete with Filter, Edit, and Share — the actions users actually reach for on every visit.

✓ Do
Export is Tertiary — visually recedes behind the view-shaping Filter.
✗ Don't
Export as Secondary competes with Filter — both have identical weight.

Edge Cases

What about "Apply" in a filter panel?

"Apply" inside a filter drawer is Primary — it's the forward action within that context. The page-level Primary (e.g. "Create New") is behind the drawer and doesn't compete. Each context gets its own hierarchy.

What about "Cancel" in a modal?

Cancel is Ghost in modals — not Secondary. It's the escape route, not an alternative path. The Secondary slot is reserved for actions like "Save as Draft" that produce a different outcome than the Primary "Save & Publish."

What about toggle buttons (e.g. Grid/List view)?

View toggles are Ghost / Icon with a visual active state (background fill on the selected option). They're not actions that change data — they're view preferences. Treat them like Ghost buttons with a persistent selected state.

What about "Archive" — is it destructive?

No. Archive is reversible — the user can restore archived items. It uses Secondary styling with a confirmation step. Destructive red is reserved for permanent, irreversible operations. Don't dilute the red signal.

Theming Boundaries

Rule: Primitive tokens are off-limits for client overrides. Status colours, typography, spacing, radius, and shadows are locked globally.
LayerOverridable?Examples
PrimitiveNo--color-green-100, --spacing-4, --radius-2, --font-size-2
SemanticOnly the 5 theming tokens--accent, --nav-bg, --nav-fg, --logo-src
ComponentOnly --btn-primary-bgAll other component tokens resolve through the cascade
✓ Do
Override the 5 theming tokens in a dedicated CSS file. Test that all interactive states (hover, active, focus) work with the new accent value.
✗ Don't
Override status colours, spacing, or typography per client. If "Cancelled" is red for one client and orange for another, the system is broken.

Composition vs. Net-New

The rule: If composing existing primitives would compromise the interaction's UX, design net-new — constrained by the token layer. Otherwise, compose.
Decision examples
Compose A confirmation dialog → Modal + Button + Text. The composition works — no UX compromise.
Compose A filter toolbar → Search Input + Select + Button. Standard components, standard layout.
Net-new An asset upload wizard with drag-drop zones, progress tracking, and metadata entry. Forcing this into a Modal composition would compromise the UX.
Net-new A rights management panel with tree-structured permissions. No existing component handles nested hierarchical selection.

Updating an Existing Component

Changes to existing components follow a lighter process than new proposals, but still require review.

Update process
1
Document the change
What's changing, why, and which surfaces are affected.
↓
2
Impact assessment
Is this a breaking change? List every place the component is used.
↓
3
Review in Design System Sync
Breaking changes require team approval. Non-breaking changes can be async.
↓
4

All three must stay in sync. A component that exists in code but not in Figma (or vice versa) is incomplete.

Versioning & Breaking Changes

Scale follows semantic versioning for communicating changes. The version number lives in the Figma file name and in the documentation header.

Change typeVersion bumpExample
Bug fix / refinementPatch (1.0.x)Fix button hover colour, adjust badge padding
New component / variantMinor (1.x.0)Add Tooltip component, add sm button size
Breaking changeMajor (x.0.0)Rename token, remove component, change component API

Breaking changes are communicated via the Design System Sync standup and documented in a changelog. Components deprecated in version N are removed no earlier than version N+1, giving teams at least one release cycle to migrate.

Rule: A component is only "in the system" when it exists in all three places: Figma, React code, and this documentation. If any one is missing, it's not a Scale component — it's a draft.
