How it stands

Neutrals accumulate wherever there isn't a single source of truth.

The product runs on the scales that were already wired in. Sand landed partway — the tokens exist, but nothing themes on them yet. Dark mode is handled per component in the tokens Unity already had. The design system defines the target. Four scales, arrived at one reasonable step at a time.

The token names stay as they are. --gray-* and --cv-* are referenced throughout the product, so we keep them and change only what they resolve to — which is what makes the move cheap, below.

The target, and why this one

One neutral: Sand on the Radix 12-step scale.

It's the natural convergence point for three practical reasons, not aesthetic ones. It inverts for dark by role — the same step means the same thing in both modes — which none of the flat product scales can do. It's already half-present in the product as the --sand-* tokens, so we're finishing a migration, not starting one. And it's what the design system already documents, so doc and product finally describe the same thing.

Why it's nearly free

A value swap, not a rename

We keep every token name — --gray-*, --cv-* — exactly as they are, so nothing in the product breaks. We only change the values they resolve to: --gray-12: #202020 → #21201c, and so on. And because Sand and Gray share the same lightness at every step, every contrast ratio is preserved — no accessibility regression, no layout shift. The grays just warm up. The names stay slightly-imperfect forever; that's cosmetic debt, not a functional one.

One honest note on the hue

We're not doing this because the warmer grays will be visible — they mostly won't. Sand and Gray are near-identical: below the just-noticeable-difference threshold (ΔE < 2.3) on seven of twelve steps, and only marginal on the dark end, shown here (left half Sand, right half Gray, touching):

Even at steps 8–12, where the difference is largest, it's ΔE ~3–4 — noticeable only side by side. That near-invisibility is the point: it's exactly why the swap carries zero visual risk. The reason to converge isn't a warmer look — it's one system instead of four, and dark mode that works by role.

What converging gets us

  • One system to maintain, not four. Doc, product light, product dark all read from the same ramp — change it once, everything follows.
  • Dark mode by role, centrally. Instead of maintaining dark per component in --cv-*, a step means the same thing in both modes and inverts on its own.
  • Doc and product finally agree. The design system stops describing a neutral the product doesn't run — which is what's made it hard to trust as a reference.
  • A cleaner footing for the FIFA DAM audit. One ratified neutral foundation with one dark convention is a calmer thing to put in front of an external accessibility review.

Worth flagging, gently: while we're split, the --sand-* tokens are easy to trip on — they resolve, so it looks like Sand is live, but building on them renders gray. Converging removes that footgun.

How we know — pulled live from the product

getComputedStyle(html).getPropertyValue('--gray-12') → #202020 (same in light & dark — ramp doesn't invert)
...getPropertyValue('--sand-1') → #fdfdfc (resolves, but nothing themes on it)
body.className → "hl-2 option5 header-brand dark"
.mask.dark-mode { background: var(--cv-g-500); } → body bg #1a1918

The --cv-* read is from one dark-mode component. Mapping the full Unity dark surface is a quick codebase pass (question 2 below), not a console check.

Decisions to make together

  1. Is Sand a decision we've committed to shipping, or has it lived as design-system intent so far? That tells us whether this is a ticket to schedule or a call to make first.
  2. What's the real blast radius of the value swap across Unity's Vue components? A short pass over --cv-* / --gray-* usage sizes it — happy to run that with whoever owns that code.
  3. Which dark trigger do we standardise on — .dark on a root, or the existing body.dark? Whatever we pick, doc and product should name the same one.
  4. Does folding this into the DAM dark-mode work make sense, given the FIFA audit is coming either way — or is it better as its own small piece first?
